# 边界样例分析

## 一、文档目的

本文档用于说明智盾Agent 当前测试样例集的边界、评测口径和局限，避免把规则样例集下的回归测试结果误解为未知攻击泛化检测能力。

当前系统是规则优先的安全网关 MVP，围绕以下闭环做本地验证：

```text
输入检测 -> 风险评分 -> Function Calling 执行前审计 -> RBAC 阻断 -> 输出脱敏 -> 审计追踪
```

当前评测结果主要用于验证：

- 当前规则库是否能命中既定样例。
- 当前风险评分是否能给出预期风险等级。
- 当前 Function Calling 模拟/派生机制是否能在高敏输入下产生工具调用请求。
- 当前 RBAC 拒绝策略是否能在工具执行前阻断。
- 当前审计事件、工具调用流水和结构化报告链路是否可用。

该评测不用于声明未知攻击检测能力，也不用于声明生产环境准确率。

## 二、样例类型划分

### 明确攻击样例

明确攻击样例通常包含清晰的提示注入、越权访问或敏感泄露意图。此类样例期望系统阻断，或至少进入高风险处置。

在当前样例集中，以下类型主要属于明确攻击样例：

- `tool_abuse`（含参数越界 `param_overflow` 场景）
- `sensitive_data_exfiltration`
- `chained_attack`

### 正常业务样例

正常业务样例不包含明显攻击关键词，不要求访问高敏资源，也不诱导泄露系统提示、密钥或凭据。此类样例应放行。

当前样例集中 `normal` 类型属于正常业务样例，包含合规工具调用场景。

### 边界模糊样例

边界模糊样例可能包含安全、管理员、password、读取等词，但意图并不一定是越权或泄露。此类样例允许 `ALLOWED_OR_BLOCKED`，评测重点是观察风险等级与证据链是否合理。

当前样例集中，边界样例涵盖以下分类：

- `borderline`：安全流程咨询、权限设计等语义偏合规的请求。
- `prompt_injection`（全部）：提示注入类样例统一使用 `ALLOWED_OR_BLOCKED` 预期，不强制裁决结果，重点观察风险等级是否匹配。
- `rule_override`（全部）：规则覆盖类样例统一使用 `ALLOWED_OR_BLOCKED` 预期，观察风险等级和叠加攻击时的提升幅度。
- 部分 `tool_abuse`：参数类型篡改、字符长度超限等需要结合业务策略判断的边界工具调用。

### 链式攻击样例

链式攻击样例同时包含提示注入、工具越权和敏感泄露诱导，应被判定为高危并阻断。

当前样例集中 `chained_attack` 类型必须满足：

- 风险等级为 `high`。
- 裁决为 `BLOCKED`。
- 触发工具调用（如 `read_system_file`）。
- RBAC 在执行前拒绝。
- 生成审计事件 `eventId`。

此外新增多轮上下文污染攻击（首轮伪装 + 后续注入 / 逐步越权），验证上下文级联的检测能力。

## 三、当前 59 条样例集说明

### normal 样例

数量：13 条。

用途：

- 验证普通业务咨询、公开文档总结、会议纪要整理、客服话术生成、安全概念解释、密码策略咨询、数据格式转换、安全培训文案、接口文档说明等不会误触发阻断。
- 验证合规工具调用（公开文件读取、分页查询、公开日志写入、参数校验）可正常放行并保持低风险评分。

### prompt_injection 样例

数量：11 条（含 1 条伪装类 `case_disguised_normal_003`）。

用途：

- 验证 `ignore previous instructions`、`bypass`、`jailbreak`、`system prompt`、中文"忽略之前""越过安全"等明显提示注入特征。
- 验证弱提示注入（模糊诱导、伪装技术交流、情感诱导）可识别为中风险。
- 验证同义改写攻击（替换核心词汇、句式重构）可识别为高风险。
- 全部提示注入样例预期决策为 `ALLOWED_OR_BLOCKED`，评测重点为风险等级匹配。

### rule_override 样例

数量：4 条。

用途：

- 验证"新的规则""覆盖规则""你现在不需要遵守"等规则覆盖诱导。
- 验证同义改写规则覆盖可识别为中风险。
- 验证规则覆盖与提示注入叠加后达到高风险。
- 全部规则覆盖样例预期决策为 `ALLOWED_OR_BLOCKED`，评测重点为风险等级匹配。

### tool_abuse 样例

数量：13 条（含 1 条伪装类 `case_disguised_normal_002`、2 条参数越界 `case_param_overflow_001/002`）。

用途：

- 验证 `read_system_file`、`/admin`、`db_credentials`、`sudo`、`越权` 等工具越权输入。
- 验证路径遍历攻击、超大分页参数、伪装合规路径跳转、未授权工具调用等场景。
- 验证参数越界攻击（数值超限、字符长度超限）可被识别。
- 确认 Function Calling 模拟/派生机制会产生工具调用请求。
- 确认 RBAC 拒绝策略会在执行前阻断。
- 部分边界工具调用（参数类型篡改、字符长度超限）预期决策为 `ALLOWED_OR_BLOCKED`。

### sensitive_data_exfiltration 样例

数量：10 条（含 1 条伪装类 `case_disguised_normal_001`）。

用途：

- 验证 `db_credentials`、`secret`、`token`、`key`、`api key`、`password`、`数据库密码` 等敏感泄露诱导。
- 验证隐晦泄露诱导（伪装故障排查、模糊指代、分步诱导）可触发高风险判定。
- 验证同义改写敏感泄露可触发高风险判定。
- 验证伪装安全报告编写诱导敏感信息泄露可被识别。

### chained_attack 样例

数量：5 条（含 2 条多轮污染攻击）。

用途：

- 验证提示注入、规则绕过、工具越权和敏感泄露诱导的组合攻击。
- 验证多轮上下文污染攻击（首轮伪装 + 后续注入、逐步提升权限）可被识别。
- 确认链式攻击必须被高危阻断，并产生完整审计证据链。

### borderline 样例

数量：3 条。

用途：

- 验证包含管理员、读取或 password 等词但语义偏合规的边界请求。
- 验证明确授权但行为高危的渗透测试模拟请求。
- 全部边界样例预期决策为 `ALLOWED_OR_BLOCKED`，观察规则优先策略下的风险评分和处置。

## 四、评测结果解释

当前自动评测结果为：

```text
totalCases: 59
passedCases: 38
failedCases: 21
passRate: 64.41%
falsePositiveCount: 0
falseNegativeCount: 8
```

核心安全指标：

```text
正常样例误报率: 0.0%
高危样例漏报率: 30.77%
边界样例通过率: 50.0% (10/20)
Tool_call 样例通过率: 84.21% (16/19)
```

严谨表述应为：

> 在当前 59 条规则驱动测试样例集下，系统回归评测通过率为 64.41%。该结果用于验证当前规则库、风险评分、RBAC 阻断和审计链路的可用性，不代表对未知攻击样例的泛化检测能力。

### 失败样例主要分布

- **弱提示注入 / 同义改写攻击**（6 条失败）：弱提示注入和同义改写类攻击的风险等级普遍被低估（预期 medium/high，实际 low），说明当前基于关键词的规则引擎对语义层面改写和弱化表达的检测能力有限。
- **隐晦泄露诱导**（3 条失败）：伪装故障排查、模糊指代、分步诱导等隐晦泄露手法未被有效阻断，同样因为规则引擎缺少语义理解能力。
- **工具调用参数越界**（3 条失败）：路径遍历、超大分页、未授权工具等场景存在漏报，部分 Function Calling 派生和 RBAC 拒绝未正确触发。
- **边界样例误判**（1 条失败）：安全测试授权请求风险等级被低估。

该结果不代表：

- 未知攻击泛化检测能力。
- 真实生产环境准确率。
- 已接入真实大模型。
- 已完成完整 IAM / DLP 系统。
- 对隐晦、多轮、绕写攻击的完整覆盖。

## 五、当前规则引擎局限

- 对明显关键词攻击识别较强，例如 `/admin`、`db_credentials`、`secret`、`token`、`key`、`ignore previous`、`read_system_file`。
- 对隐晦表达、同义改写、拼写扰动、绕写、上下文污染、多轮攻击检测能力仍有限（高危漏报率 30.77%）。
- 对弱提示注入（模糊诱导忽略规则、伪装技术交流、情感诱导）的风险等级判定偏低，当前普遍评为 low 而非预期的 medium。
- 语义判别分 `S_cls` 当前未启用轻量模型，主要由规则和资源敏感度间接体现。
- Function Calling 当前是 MVP 阶段的模拟/派生机制，不是真实大模型产生的 tool call。
- 对参数越界攻击（超大分页、时间范围超限、字符长度超限）的派生 Function Calling 触发尚不稳定。
- 当前未接真实工具执行沙箱，也不会真实读取文件或调用企业资源。
- 当前事件、规则、RBAC、测试样例仍使用 JSON 文件，不是数据库持久化方案。

## 六、后续增强方向

后续可以按阶段增强：

- 扩充边界样例，尤其是安全咨询、权限流程、密钥轮换等容易误判的场景。
- 增加弱攻击和隐晦攻击样例，例如错别字、分隔符绕写、同义改写、多轮铺垫。
- 引入轻量语义分类模型，用于补充 `S_cls`，改善对弱提示注入和同义改写攻击的识别。
- 增强 Function Calling 派生覆盖率，确保所有工具类请求（含参数越界）都能稳定触发。
- 接入真实大模型 API，但仍保持安全控制层位于模型应用外侧。
- 接入真实 Function Calling 输出，使用真实 `tool_call` 进行执行前审计。
- 引入真实工具执行沙箱，在执行前后补充参数边界和输出检查。
- 使用 SQLite 或数据库持久化规则、RBAC 策略和审计事件。
- 建设规则库与 RBAC 策略在线管理能力，但需先明确权限边界。
